指令：构建文档处理中心 (Document Processing Hub)

**任务：**
在 `app/(dashboard)/knowledge/[lib_id]/page.tsx` 开发知识库详情页。该页面主要负责展示当前库下的文档列表，并提供上传新文档、配置切片参数的功能。

**详细要求：**

#### 1. 页面结构设计

* **面包屑导航：** 顶部显示 `知识库 / [库名称]`，点击可返回列表页。
* **双区域布局：**
* **上方/侧边：** 文档上传区。
* **主体：** 已上传文档的状态列表。



#### 2. 文档上传与切片配置流（核心业务）

* **第一步：文件选择：** 实现一个拖拽上传区域。用户选择文件后，立即调用文件上传接口。
* **第二步：配置确认：** 文件上传成功后，**不要直接提交入库**。弹出一个配置面板，显示该文件的基本信息，并允许用户设置：
* **分段长度 (chunk_size)：** 数字输入框，默认 500。
* **分段重叠 (chunk_overlap)：** 数字输入框，默认 50。


* **第三步：任务提交：** 用户确认参数后，调用 `SysDocuments` 创建接口，将文件与当前 `lib_id` 关联。

#### 3. 文档状态列表 (Status Tracking)

* 使用 `Table` 组件展示文档，包含以下列：
* **文件名：** 对应 `origin_name`。
* **切片参数：** 显示 `chunk_size / chunk_overlap`。
* **分块数量：** 对应 `chunk_count`。
* **处理状态：** 对应 `processing_status`。
* **操作：** 删除按钮。


* **状态渲染逻辑：**
* `pending`: 灰色 Badge，显示“排队中”。
* `processing`: 蓝色进度条或闪烁动画，显示“解析中”。
* `completed`: 绿色 Badge，显示“已完成”。
* `failed`: 红色 Badge，鼠标悬停时通过 `Tooltip` 显示 `error_msg`。



#### 4. 自动刷新逻辑 (Polling)

* **实时性：** 如果列表中存在状态为 `pending` 或 `processing` 的文档，前端需每隔 3 秒自动请求一次文档列表接口，实现状态自动更新，直到所有任务结束。

#### 5. 交互优化

* **Loading 状态：** 上传过程中显示进度条。
* **空状态：** 如果当前库无文档，展示“尚未上传文档”的插画及上传引导。
